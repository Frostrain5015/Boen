# Boen MVP 收口计划

版本：v0.3.3 审计后

目标：在不包含会员支付接入的前提下，使产品具备面向有限真实学生用户开放的安全性、可恢复性、质量保障和运营基础。

## MVP 定义

首发范围内的已登录学生能够稳定完成：学习对话、练习或出卷、作答和批改、错题复习；发生弱网、服务异常或误操作时，用户数据可恢复、问题可定位、运营人员可处理。

首发不以“全学科”作为承诺。每个开放年级、学科和教材版本必须有明确的可用范围与核验状态。

## 放行标准

| 阶段 | 是否可对外开放 | 条件 |
| --- | --- | --- |
| 当前状态 | 否 | 功能闭环存在，但权限隔离、数据恢复、自动化质量门禁和可观测性不足。 |
| 封闭测试 | 可以，限白名单 | 完成全部 P0；课程范围明确；使用真实设备和弱网场景验收。 |
| 公开 MVP | 可以 | 完成 P0、主要 P1，并稳定运行一个封闭测试周期。 |

## P0：公开前必须完成

### 1. 关闭全局模型切换越权

**问题**：`POST /api/model/switch` 仅要求登录，但会切换进程共享模型。任意用户可影响所有用户的模型、成本和服务质量。

**实施**：

- 移除面向普通用户的模型切换接口，或改为管理员专用配置。
- 将模型选择收敛为服务端受控策略；如确需用户偏好，必须按用户或请求隔离，禁止修改全局运行时实例。
- 对模型、出卷、OCR 等高成本请求增加用户、IP 和并发三级限流。

**验收**：普通用户无法改变其他用户的模型；压测或手工并发验证中，模型配置和额度彼此隔离。

### 2. 补齐未成年人数据和用户权利

**问题**：产品面向中小学生，但缺少可访问的条款、隐私说明、AI/OCR 数据处理说明、删除/导出机制和数据保留规则。

**实施**：

- 发布服务条款、隐私政策、未成年人使用说明和 AI 生成内容说明，并在登录和首次使用时可访问。
- 记录同意版本、时间和用户标识。
- 提供账户数据导出、账户删除和运营侧处理流程。
- 明确对话、错题图片、考试记录、日志和备份的保留期限及删除边界。

**验收**：测试账户可查看政策、导出自身数据并提交删除请求；运营可查询并处理请求。

### 3. 建立自动化质量门禁

**问题**：当前没有统一的 `test`、`lint`、`typecheck`、端到端测试和 CI 入口，发布主要依赖人工构建。

**实施**：

- 在根目录提供统一的 `lint`、`typecheck`、`test`、`test:e2e` 和 `check` 命令。
- 在 CI 中强制执行类型检查、前后端构建和关键测试。
- 最低测试集覆盖：认证与订阅权限、模型切换权限、SSE 事件顺序和中断、考试 schema 与批改、错题与图片所有权、删除全部错题后的空态。
- 增加生产部署后的健康检查冒烟测试。

**验收**：任意 PR 或发布提交未通过检查不能合并或部署；关键路径测试可在干净环境重复运行。

### 4. 数据迁移、备份与恢复

**问题**：SQLite schema 主要由运行时建表和条件升级维护，缺少版本化迁移、回滚策略和自动恢复演练。

**实施**：

- 引入版本化 migration 和 schema 版本表；将 seed 与 schema 升级分离。
- 对 SQLite 数据库和用户上传图片做加密或受控访问的定期备份。
- 在部署前运行 migration 预检，在失败时阻止发布。
- 建立恢复 runbook，并实际执行一次恢复演练。

**验收**：可从空环境和历史备份分别恢复到目标版本；恢复后能登录、查看历史考试和错题资产。

### 5. 可观测性、告警和生产运行时

**问题**：健康接口偏存活检查，缺乏请求关联、错误采集、模型/OCR 成本与失败率指标；云端 Node 版本与部分依赖 engine 要求不一致。

**实施**：

- 输出结构化日志和请求 ID，覆盖认证、SSE、模型调用、OCR、出卷、TikZ 和数据库异常。
- 接入错误监控；记录模型调用耗时、token/成本、失败率、队列长度及限流命中。
- 将 health 拆分为 liveness 与 readiness；readiness 检查数据库、可写数据目录和必要配置。
- 统一本地、CI 与服务器 Node LTS 版本，并将其固定在部署环境。
- 配置异常率、连续健康检查失败、磁盘不足、备份失败和成本异常告警。

**验收**：可根据一次失败请求定位用户、请求、依赖调用和错误；模拟数据库或模型不可用时 readiness 失败并产生告警。

### 6. 成本、滥用与边界保护

**实施**：

- 将 CORS 收敛为正式 Web 域名和必要开发域名。
- 为聊天、出卷、OCR、认证代理和 TikZ 设置请求体、频率、并发、配额与超时边界。
- 高成本任务使用队列，返回可理解的排队、取消和稍后重试状态。
- 为管理员操作、异常认证请求和额度耗尽提供审计日志。

**验收**：同一用户或 IP 的异常请求不会拖垮服务；额度耗尽、排队和超时均有稳定 API 与前端提示。

## P1：封闭测试期间完成

### 7. 弱网恢复和幂等提交

- 为聊天 SSE 定义事件 ID、重放或安全恢复策略，避免断线重复提问或丢失最终回答。
- 为考试答案启用本地草稿、自动保存、刷新恢复和提交幂等键。
- 明确网络恢复后哪些操作会自动同步，哪些需要用户确认。

**验收**：在请求中断、刷新页面、断网后恢复网络等场景中，不重复生成、不丢失已提交答案，且状态可解释。

### 8. 课程知识库发布机制

- 建立公开的首发课程范围矩阵：年级、学科、教材版本、已核验状态和已知限制。
- 对每个已开放范围保留来源、人工核验记录和种子数据验证报告。
- 支持快速下线存在质量问题的学科或知识点范围。

**验收**：产品页面、README 和种子数据范围一致；运营人员可以回答“该知识点是否可用、依据是什么”。

### 9. 真机、可访问性与学生体验

- 在 iOS Safari、Android Chrome、低分辨率和较慢网络下验收聊天、上传、长试卷和错题详情。
- 覆盖键盘操作、屏幕阅读器标签、焦点状态、色彩对比和减少动态效果。
- 加入用户可见的反馈/客服入口及内容问题上报路径。

**验收**：定义的设备矩阵无阻断问题；关键操作可通过键盘完成；学生能提交问题并得到可追踪编号。

### 10. 订阅权益的人工运营流程

会员支付不在本计划范围内，但订阅权益仍需具备人工开通、续期、撤销、申诉和纠错流程，避免支付接入前的测试用户被错误锁定或错误放行。

**验收**：运营人员可安全调整测试用户权益，所有变更可追溯。

## 建议执行顺序

1. P0-1、P0-6：先封堵权限和成本风险。
2. P0-3、P0-4：建立可重复发布和可恢复基础。
3. P0-5：补齐监控、告警和运行时一致性。
4. P0-2：上线前完成用户权利和未成年人数据闭环。
5. 建立课程范围矩阵并开始白名单封闭测试。
6. 在封闭测试中完成 P1-7 至 P1-10，再评估公开 MVP 放行。

## 发布前最终检查清单

- [ ] 普通用户无法修改全局模型或其他用户配置。
- [ ] CI 全绿：类型检查、构建、单测、E2E 冒烟。
- [ ] 生产数据库与上传资产已备份，并完成恢复演练。
- [ ] 生产健康检查、错误监控、成本告警和日志检索可用。
- [ ] 条款、隐私、AI/OCR 说明、删除与导出流程可访问。
- [ ] 首发课程范围和质量状态已公开且与数据一致。
- [ ] 真机和弱网验收通过。
- [ ] 订阅权益人工处理流程已验证。
